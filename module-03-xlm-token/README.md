## 1. What is XLM (Lumens)?
 
Lumens (XLM) are the native currency of the Stellar network. XLM holds three properties that make it unique among all assets on the network:
 
| Property             | XLM                          | Other Stellar Assets        |
|----------------------|------------------------------|-----------------------------|
| Requires an issuer   | No - created at network genesis | Yes - every asset has an issuer |
| Requires a trustline | No                           | Yes - receiver must opt in  |
| Used to pay fees     | Yes - exclusively            | No                          |
 
XLM serves three essential functions on the network:
 
- **Transaction fees** - all fees are paid in XLM, without exception
- **Minimum balance** - every account must hold a minimum amount of XLM to exist on the ledger
- **Smart contract rent** - Soroban contract storage is paid in XLM
> Source: [Lumens (XLM) - developers.stellar.org](https://developers.stellar.org/docs/learn/fundamentals/lumens)
 
 
## 2. Supply - Origin and the 2019 Burn
 
### Initial supply
 
When the Stellar network launched in 2014, exactly **100,000,000,000 XLM** (100 billion) were created. This was the only original issuance. Unlike Bitcoin or Ethereum, Stellar has no mining, so XLM cannot be produced through block rewards.
 
### Network inflation (2014 - 2019)
 
For the first five years of Stellar's existence, the protocol included an annual inflation mechanism that increased the total supply by 1% each year. This mechanism was removed by validator vote on **October 28, 2019**.
 
The total XLM generated by inflation before removal was `5,443,902,087 XLM`. Adding this to the original supply, the maximum historical supply ever reached was approximately **105.4 billion XLM**.
 
### The 2019 burn
 
On **November 4, 2019**, the SDF reduced its lumen holdings by sending `55,442,095,285 XLM` to account [`GALA...LUTO`](https://stellar.expert/explorer/public/account/GALAXYVOIDAOPZTDLHILAJQKCVVFMD4IKLXLSZV5YHO7VY74IWZILUTO) - an address with no signers, making the funds permanently inaccessible.
 
This single action reduced the total supply from ~105B to approximately **~50 billion XLM**, where it has remained since.
 
> Source: [SDF's Next Steps (2019 Burn Post) - stellar.org](https://www.stellar.org/blog/sdfs-next-steps)
 
### Fee pool
 
Transaction fees collected from all operations accumulate in a special **Fee Pool** - a locked account that no one can access. These XLM are not redistributed, not burned, and not given to validators. They simply accumulate. Given how low fees are, the fee pool grows very slowly.

## 4. Transaction Fees - The Anti-Spam Mechanism
 
Stellar requires a fee for every transaction to reach the ledger. The purpose is not to generate revenue - it is to prevent spam and to prioritize transactions during network congestion. All fees are paid in XLM.
 
### The Stroop - smallest unit of XLM
 
```
1 stroop = 0.0000001 XLM  (one ten-millionth of a lumen)
```
 
### Inclusion fee (classic transactions)
 
For transactions that do not execute smart contracts (payments, account creation, trustlines, etc.):
 
```
inclusion fee = number of operations x effective base fee
```
 
The **minimum effective base fee** is **100 stroops per operation** (= 0.00001 XLM). A single-operation payment costs at minimum 0.00001 XLM - a fraction of a cent at any realistic XLM price.
 
**Important behavior:** when you set a base fee, you are declaring the **maximum** you are willing to pay per operation. Under normal (non-surge) conditions, you only pay the network minimum of 100 stroops, even if you declared a higher maximum.
 
A single transaction can contain **up to 100 operations** (classic transactions only). This means the maximum cost of a 100-operation transaction at minimum fee is still only 0.001 XLM.
 
### Where do fees go?
 
All collected fees go into the locked Fee Pool. They do not go to validators, to the SDF, or to any account. There are no fee-based incentives for running a validator on Stellar.
 
> Source: [Fees, Resource Limits, and Metering - developers.stellar.org](https://developers.stellar.org/docs/learn/fundamentals/fees-resource-limits-metering)
 
 
## 5. Surge Pricing
 
When the number of operations submitted to a ledger exceeds the network capacity (currently **1,000 operations** for classic transactions), the network enters **surge pricing mode**.
 
In surge pricing:
- Transactions compete for ledger space by fee bid
- Higher fee bids are included first
- All included transactions pay the fee of the lowest bid in the included set
- Transactions that do not make it are pushed to the next ledger or discarded if they have been waiting too long
### Example
 
Five transactions compete, but the ledger only has space for four:
 
```
Fee bids:  [2, 3, 4, 4, 5] XLM
Space for: 4 transactions
Excluded:  the transaction bidding 2 XLM
All 4 included transactions pay: 3 XLM (the lowest bid in the included set)
```
 
If all five fit (non-surge), each pays the network minimum of 100 stroops.
 
When multiple transactions share the same fee and not all can fit, the selection is random.
 
**Recommendation:** use `ledger bounds` or `time bounds` on transactions so they either make it to the ledger or fail cleanly, rather than staying in the queue indefinitely.
 
> Source: [Fees, Resource Limits, and Metering - developers.stellar.org](https://developers.stellar.org/docs/learn/fundamentals/fees-resource-limits-metering)
 
 
## 6. Fee-Bump Transactions
 
If a transaction fails to make it to the ledger due to insufficient fee, it can be "re-paid" using a **Fee-Bump Transaction** - a special transaction that wraps the original and attaches a higher fee.
 
Rules:
- The fee-bump must offer at least **10x** the original transaction's fee to replace it in the queue
- The fee-bump can be signed and submitted by a different account than the original transaction's source account
- This is useful for applications that want to absorb fee costs on behalf of users
Fee-bump transactions are also the mechanism that allows smart contract transactions to have a second operation (the fee-bump itself adds one operation to the smart contract transaction).
 
> Source: [Fees, Resource Limits, and Metering - developers.stellar.org](https://developers.stellar.org/docs/learn/fundamentals/fees-resource-limits-metering)
 
 
## 7. Base Reserve - The Minimum to Exist
 
Every Stellar account must maintain a **minimum balance** of XLM at all times to exist on the ledger. This prevents the ledger from filling up with empty or abandoned accounts.
 
### Base Reserve unit
 
```
1 Base Reserve = 0.5 XLM  (current value, set by validator vote)
```
 
Validators can vote to change the base reserve, but this is uncommon and expected to happen only every few years.
 
### Minimum balance calculation
 
Every account requires a base of **2 Base Reserves (1 XLM)** simply to exist. Each additional item stored on the ledger (called a **subentry**) adds one Base Reserve to the minimum:
 
| Subentry Type          | Additional Base Reserves |
|------------------------|--------------------------|
| Trustline (asset)      | 1 (0.5 XLM)             |
| Trustline (pool share) | 2 (1.0 XLM)             |
| Active offer           | 1 (0.5 XLM)             |
| Additional signer      | 1 (0.5 XLM)             |
| Data entry             | 1 (0.5 XLM)             |
| Claimable balance      | 1 per claimant (0.5 XLM)|
 
Maximum subentries per account: **1,000**.
 
### Calculation example
 
An account with one trustline, two active offers, and one claimable balance with one claimant:
 
```
Base account:       2 base reserves = 1.0 XLM
+ 1 trustline:      1 base reserve  = 0.5 XLM
+ 2 offers:         2 base reserves = 1.0 XLM
+ 1 claimable bal:  1 base reserve  = 0.5 XLM
                                    ----------
Minimum balance:                      3.0 XLM
```
 
When a subentry is closed (for example, a trustline is removed), the corresponding Base Reserve is returned to the account's available balance.
 
> Source: [Lumens - developers.stellar.org](https://developers.stellar.org/docs/learn/fundamentals/lumens)
 
 
## 8. Sponsored Reserves
 
For applications that want to onboard users without requiring them to hold XLM, Stellar provides a **Sponsored Reserves** mechanism: a sponsoring account pays the Base Reserve on behalf of another account.
 
This enables:
- Creating accounts with zero initial XLM for end users
- Paying trustline costs on behalf of users
- Absorbing all minimum balance requirements at the application layer
A sponsored reserve is created using a "sandwich transaction" with three operations:
1. `Begin Sponsoring Future Reserves` - initiated by the sponsoring account
2. The operation being sponsored (e.g., account creation, trustline)
3. `End Sponsoring Future Reserves` - accepted by the sponsored account, requires its signature
> Source: [Sponsored Reserves - developers.stellar.org](https://developers.stellar.org/docs/build/guides/transactions/sponsored-reserves)
 
 
## 9. Smart Contract Fees (Soroban)
 
Since Protocol 20 (February 2024), Soroban smart contracts on Stellar use a different fee model with two components:
 
```
Transaction Fee = Resource Fee + Inclusion Fee
```
 
### Resource fee
 
Covers the actual cost of executing the contract. Calculated from the following resource dimensions:
 
| Resource              | Description                                                    |
|-----------------------|----------------------------------------------------------------|
| CPU Instructions      | Number of CPU instructions executed by the contract            |
| Ledger Entry Accesses | Reading or writing any single ledger entry                     |
| Ledger I/O            | Bytes read from or written to the ledger                       |
| Transaction Size      | Size of the submitted transaction in bytes                     |
| Events & Return Value | Size of contract events and the top-level function return value|
| Rent                  | Payment for ledger entry TTL extensions and size increases     |
 
The resource fee has two sub-components:
- **Non-refundable fees** - CPU instructions, read/write bytes, bandwidth. Charged regardless of execution outcome.
- **Refundable fees** - rent, events, return value. Charged upfront and refunded based on actual usage.
### Rent
 
Smart contract data does not require Base Reserves. Instead, it pays **rent** based on entry size and how long it should remain on the ledger before being archived. Three storage types have different rent requirements: `Persistent`, `Temporary`, and `Instance`.
 
### Ledger limits for smart contracts
 
Smart contract transactions compete separately from classic transactions:
 
| Transaction Type | Ledger Limit         |
|------------------|----------------------|
| Classic          | 1,000 operations     |
| Soroban          | 100 transactions     |
 
The tighter limit for Soroban means surge pricing is more frequent for smart contract transactions. Plan fee bidding strategies accordingly.
 
> Source: [Fees, Resource Limits, and Metering - developers.stellar.org](https://developers.stellar.org/docs/learn/fundamentals/fees-resource-limits-metering)
